## [ERR-20250317-001] 年份判断错误

**记录时间**: 2025-03-17T21:30:00+08:00
**优先级**: high
**状态**: resolved
**领域**: workflow

### 摘要
记账时将2026年的支出错误地记录到了2027年文件

### 错误详情
用户要求记录"晚饭40"，我读取账本后发现2027.md状态为active，但完全空白。实际上今天日期是2026-03-17，应该记录到2026.md。我在2027.md添加了记录并更新了统计数据，导致年份错误。

### 根本原因分析

1. **依赖文件状态元数据，未交叉验证**
   - 看到2027.md的YAML frontmatter中`status: active`就认定这是当前年度
   - 没有检查文件内容是否为空白
   - 没有验证当前实际日期

2. **数据不一致问题**
   - 2026.md状态是archived但有今天的记录
   - 2027.md状态是active但完全空白
   - 两个文件的状态标记与实际使用情况矛盾

3. **缺乏上下文验证**
   - 用户说"今天还是记录了的"时，我没有立即检查2026.md今天的记录
   - 直到用户指出才意识到年份错误

### 修复措施
1. 撤销2027.md的所有更改，恢复为空白模板
2. 在2026.md的03-17追加晚饭40记录
3. 更新2026.md的3月统计数据（总支出 3092→3132）
4. 更新年度总览数据

### 预防建议
1. 记账前先问用户或确认当前年份
2. 检查"active"年份文件是否有当月记录，如果没有则确认是否选错年份
3. 看到空白active文件时应该警觉

### 修复脚本（2025-03-17）
1. `new_year.py` 新建年度默认 `status: template` 而非 `active`
2. 新增 `--activate` 命令：自动归档旧年度 + 激活新年度
3. 新增 `--archive` 命令：单独归档指定年份

**新用法**:
```bash
# 创建下一年模板（状态为template）
python scripts/new_year.py 2028

# 跨年时激活（自动归档旧年度）
python scripts/new_year.py --activate 2028
```
